# Код-ревью: менеджер транзакций и использование в репозитории

## Текущая картина

- **TxManager** работает с `*sql.Tx` (stdlib): `BeginTx` → передаёт в `fn(ctx, tx)`.
- **Репозиторий** внутри использует **sqlx**: reader — `*sqlx.DB` (readPool), writer — `*sqlx.DB` (writePool) для маппинга по db-тегам.
- В **reader** при вызове внутри транзакции приходится оборачивать `*sql.Tx` в `*sqlx.Tx` с `readPool.Mapper` (в `sqlxGetContext`).
- В **writer** в `selectByID` создаётся `*sqlx.Tx` с `writePool.Mapper` прямо в методе.

Итог: снаружи везде `*sql.Tx`, внутри — sqlx, обёртка делается в двух местах по-разному.

---

## Проблемы

### 1. Несогласованность обёртки `*sql.Tx` → `*sqlx.Tx`

| Место | Поведение |
|-------|-----------|
| **reader/get_one.go** | Хелпер `sqlxGetContext`: при `tx != nil` создаётся `&sqlx.Tx{Tx: tx, Mapper: r.readPool.Mapper}` и вызывается `GetContext`. |
| **writer/select.go** | В методе напрямую: `sxTx := &sqlx.Tx{Tx: tx, Mapper: r.writePool.Mapper}` (без проверки `tx != nil`). |

В reader логика «транзакция или пул» централизована в одном хелпере; в writer обёртка размазана по коду и не выделена в общий метод. Контракт разный: в reader `tx` опционален, в writer подразумевается «всегда в транзакции», но тип один — `*sql.Tx`.

### 2. Два «исполнителя» в одном стеке

В pgx обычно один тип на весь путь: `pgx.Tx` (или пул) передаётся в репо, и репо с ним работает. У вас: API репо — `*sql.Tx`, реализация — sqlx (пулы и обёрнутый Tx). Из-за этого в каждом месте, где нужно выполнить запрос «в транзакции», кто-то должен знать про Mapper и создавать `sqlx.Tx`. Это и даёт ощущение непоследовательности.

### 3. Риск рассинхронизации Mapper

Reader использует `readPool.Mapper`, writer — `writePool.Mapper`. Сейчас оба пула создаются из одного `*sql.DB` с одним драйвером, но если в будущем пулы разойдутся или появятся кастомные Mapper’ы, логика обёртки в двух пакетах может разъехаться.

---

## Что можно сделать

### Вариант A. Минимальные правки (консистентность в рамках текущего API)

Цель: один и тот же паттерн «обёртка Tx → sqlx» и один хелпер на слой, без смены типа в API.

1. **Reader** — уже ок: один хелпер `sqlxGetContext(ctx, tx, ...)` с веткой `tx != nil` / иначе pool.
2. **Writer** — вынести обёртку в один хелпер и использовать его везде, где внутри транзакции нужен sqlx:
   - например, `writerTx(tx *sql.Tx) *sqlx.Tx { return &sqlx.Tx{Tx: tx, Mapper: r.writePool.Mapper} }`;
   - в `selectByID` вызывать `r.writerTx(tx).GetContext(...)`.

Так обёртка `*sql.Tx` → `*sqlx.Tx` в writer делается в одном месте, и стиль с reader выравнивается (в обоих слоях есть явный «исполнитель» для GetContext).

### Вариант B. Транзакции через sqlx (как с pgx — один тип по всему стеку)

Цель: репозиторий и сервисы работают с одним типом транзакции, без обёрток внутри репо.

1. **TxManager** начинает транзакцию через **sqlx** и передаёт в колбек уже `*sqlx.Tx`:
   - например, зависимость не от `database.DBProvider`, а от интерфейса с `BeginTxx(ctx, opts) (*sqlx.Tx, error)`;
   - или расширить провайдер: давать `*sqlx.DB` (для MySQL — один раз оборачивать `sql.DB` в `sqlx.NewDb(db, "mysql")` при старте приложения).
2. В **executeTransaction** вызывать `db.BeginTxx(ctx, opts)` (или аналог), в `fn` передавать `*sqlx.Tx`.
3. **Интерфейсы репозитория** заменить `tx *sql.Tx` на `tx *sqlx.Tx` (или интерфейс с `GetContext`/`ExecContext`/`SelectContext`, если хотите слабую связь).
4. **Reader**: при `tx != nil` использовать `tx.GetContext` (tx уже `*sqlx.Tx`), при `tx == nil` — `readPool.GetContext`. Обёртка в репо не нужна.
5. **Writer**: везде использовать переданный `tx` для `ExecContext`/`GetContext`; создание `sqlx.Tx` в `selectByID` убрать.

Плюсы: один тип транзакции по стеку, нет ручного оборачивания в репо, поведение ближе к типичному использованию pgx. Минусы: нужно менять интерфейс TxManager и все вызовы (сервисы, адаптеры), а также способ передачи БД в TxManager (sqlx.DB или новый интерфейс).

#### Вариант B не ломает проект: MySQL и пулы сохраняются

- **MySQL** — не меняется. sqlx работает поверх `database/sql`; драйвер MySQL по-прежнему регистрируется и используется. Меняется только то, откуда берётся транзакция: `*sql.DB.BeginTx` → `*sqlx.DB.BeginTxx`. Оба метода используют один и тот же пул и то же подключение к MySQL.
- **Пул соединений** — один и тот же. Сейчас: `Pool` хранит `*sql.DB`; reader/writer делают `sqlx.NewDb(db, "mysql")` и получают обёртку над **тем же** `*sql.DB`. При варианте B: пул по-прежнему один `*sql.DB` (внутри `Pool` или в месте сборки приложения). Один раз делаем `sqlxDB := sqlx.NewDb(pool.DB(), "mysql")` и передаём этот `*sqlx.DB` в TxManager и в конструкторы репозиториев. Новых соединений или второго пула не появляется — `sqlx.DB` только оборачивает существующий `*sql.DB`.
- **Сборка (wire)** — меняется только способ передачи зависимости в TxManager: вместо `txmanager.New(pool, ...)` вызываем `txmanager.New(sqlxDB, ...)`, где `sqlxDB = sqlx.NewDb(pool.DB(), "mysql")`. Репозитории могут принимать тот же `*sqlx.DB` как readPool/writePool (один инстанс на оба), либо по-прежнему получать `*sql.DB` и внутри делать `sqlx.NewDb` — как удобнее. В любом случае под капотом остаётся один пул и один драйвер MySQL.

Итог: вариант B — это замена типа транзакции в API (`*sql.Tx` → `*sqlx.Tx`) и одна обёртка `*sql.DB` в `*sqlx.DB` при инициализации. Ни драйвер, ни пул, ни количество соединений не меняются.

#### Полный переход на sqlx.DB (текущее состояние)

Pool хранит и отдаёт `*sqlx.DB`: при создании пула вызывается `sqlx.NewDb(sqlDB, driverName)` один раз внутри `connectingpool.New`. При bootstrap передавайте один и тот же инстанс везде:

- `txmanager.New(pool.SqlxDB(), "txmanager")`
- `reader.NewRepository(pool.SqlxDB())`, `writer.NewRepository(pool.SqlxDB())`

Отдельно создавать `sqlx.NewDb` в main/wire не нужно — Pool уже это делает. Метод `Pool.DB()` оставлен для совместимости (возвращает `*sql.DB` того же пула).

---

## Рекомендация

- **Краткосрочно**: сделать **вариант A** — в writer ввести хелпер для получения `*sqlx.Tx` из `*sql.Tx` и использовать его в `selectByID` (и при появлении других мест с такой же обёрткой). Так код станет консистентным без смены контрактов.
- **Долгосрочно**: если планируете развивать этот слой и держать его «как с Postgres», имеет смысл перейти на **вариант B** — TxManager отдаёт `*sqlx.Tx`, репозиторий принимает его и не знает про `*sql.Tx`. Тогда текущее «выглядит не консистентно» исчезнет на уровне типов и одного способа работы с транзакцией.
