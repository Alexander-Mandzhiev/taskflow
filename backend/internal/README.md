# Модули приложения (соответствие ТЗ)

Два домена: **Identity** и **Teams**. Ниже — привязка к API и требованиям ТЗ.

---

## Identity

### account
**ТЗ:** Регистрация и аутентификация.

| API | Метод | Ответственность |
|-----|--------|------------------|
| Регистрация | `POST /api/v1/register` | Создание пользователя |
| Вход | `POST /api/v1/login` | Аутентификация, выдача JWT |

Реализация: регистрация, логин, выдача/проверка JWT.

### user
**ТЗ:** Пользователь как сущность (таблица users); нужен для приглашений и назначения задач.

- Поиск пользователя по email (для invite в команду).
- Данные пользователя по ID (профиль, assignee в задачах).
- Не дублирует «логин» — только сущность User и её чтение/поиск.

## Teams

### team
**ТЗ:** Управление командами.

| API | Метод | Ответственность |
|-----|--------|------------------|
| Создать команду | `POST /api/v1/teams` | Создание команды, создатель = owner |
| Список команд пользователя | `GET /api/v1/teams` | Команды, где пользователь в team_members |
| Пригласить в команду | `POST /api/v1/teams/{id}/invite` | Только owner/admin; вызов email-сервиса (circuit breaker по ТЗ) |

Роли: owner, admin (и member) — из `team_members.role`.

**Доступ и группировка (дашборд):** Пользователь видит только команды, в которых состоит (фильтр по `user_id` в таблице **team_members**, не в teams). У каждой команды — свои задачи (`tasks.team_id`). Выборка: (1) команды пользователя — JOIN `teams` с `team_members` по `team_members.user_id = :user_id`; (2) задачи — по `team_id` только из списка команд пользователя; ответ можно отдавать с группировкой по командам (например, `GET /api/v1/teams` с вложенным списком задач по каждой команде или отдельно `GET /api/v1/tasks?team_id=...` с проверкой членства).

**Сложный SQL (a) по ТЗ:** для каждой команды — название, количество участников, количество задач в статусе done за последние 7 дней (JOIN 3+ таблиц + агрегация). — Реализовать в этом пакете (или в отдельном report, если удобнее).

### task
**ТЗ:** Управление задачами.

| API | Метод | Ответственность |
|-----|--------|------------------|
| Создать задачу | `POST /api/v1/tasks` | Только член команды |
| Список задач | `GET /api/v1/tasks?team_id=&status=&assignee_id=` | Фильтрация + пагинация (LIMIT/OFFSET или cursor) |
| Обновить задачу | `PUT /api/v1/tasks/{id}` | Проверка прав; запись в audit при изменении |

**Оптимизация по ТЗ:** кеш Redis — список задач команды, TTL 5 мин. Индексы MySQL, connection pooling, пагинация на уровне БД.

**Сложные SQL по ТЗ:**  
- **(b)** Топ-3 пользователя по количеству созданных задач в каждой команде за месяц (подзапрос/оконная функция).  
- **(c)** Задачи, где assignee не является членом команды этой задачи (валидация целостности).  

Реализовать в этом пакете (или вынести отчёты в отдельный подпакет task/report).

### audit
**ТЗ:** История изменений задач.

| API | Метод | Ответственность |
|-----|--------|------------------|
| История задачи | `GET /api/v1/tasks/{id}/history` | Чтение из task_history |

Запись в task_history при обновлении задачи (из task при PUT).

### comment
**ТЗ:** В БД есть task_comments; в списке API явно не указаны эндпоинты комментариев.

Оставить модуль: таблица и связи есть, при необходимости добавить `GET/POST /api/v1/tasks/{id}/comments`.

## Вне доменов (общее по ТЗ)

- **Конфиг:** YAML/ENV — отдельный слой (config), не доменный модуль.
- **Rate limiting:** 100 запросов/мин на пользователя — middleware/transport.
- **Circuit breaker:** при вызове внешнего сервиса (например, мок email для приглашений) — обёртка в pkg или internal при вызове из team.
- **Graceful shutdown:** в cmd/main или слой приложения.
- **Prometheus:** метрики (запросы, ошибки, время ответа) — middleware/transport.
- **Redis:** кеш списка задач команды — слой кеша, используемый из task.
- **Индексы MySQL:** добавить миграции (индексы по team_id, status, assignee_id, датам и т.д.) в db/migration.

---

## Зависимости между модулями

- **Identity** не зависит от Teams.
- **Teams** использует только идентификатор пользователя (из JWT/Identity); при invite — поиск user по email (identity/user).
- Запись в audit вызывается из task при обновлении задачи.
